## üîÑ Flujo de Trabajo T√≠pico

### Para un Employer:

1. **Crear propuesta** ‚Üí INSERT en `proposals`
2. **Agregar hitos** ‚Üí INSERT en `milestones`
3. **Recibir aplicaciones** ‚Üí Freelancers crean en `proposal_applicants`
4. **Seleccionar freelancer** ‚Üí UPDATE `proposals.selected_freelancer_id` y `proposal_applicants.status = 'aceptado'`
5. **Iniciar chat** ‚Üí INSERT en `chat_threads` vinculado a la propuesta
6. **Completar hitos** ‚Üí UPDATE `milestones.status = 'completado'`
7. **Finalizar trabajo** ‚Üí UPDATE `proposals.status = 'completada'`

### Para un Freelancer:

1. **Ver propuestas** ‚Üí SELECT en `proposals WHERE status = 'publicada'`
2. **Guardar propuesta** ‚Üí INSERT en `saved_proposals`
3. **Aplicar a propuesta** ‚Üí INSERT en `proposal_applicants`
4. **Recibir aceptaci√≥n** ‚Üí `proposal_applicants.status = 'aceptado'`
5. **Chatear con employer** ‚Üí INSERT en `chat_messages`
6. **Completar hitos** ‚Üí UPDATE `milestones.status`
7. **Actualizar portafolio** ‚Üí INSERT en `portfolio_entries`

## üîê Pol√≠ticas de Seguridad (RLS)

Aseg√∫rate de tener Row Level Security (RLS) activado en todas las tablas para:

- **profiles**: Los usuarios pueden ver todos los perfiles, pero solo editar el suyo
- **proposals**: P√∫blicas para lectura, solo el employer puede editar las suyas
- **proposal_applicants**: Freelancers ven las suyas, employers ven las de sus propuestas
- **milestones**: Solo visibles para el employer y el freelancer asignado
- **chat_threads/messages**: Solo los participantes (user_a y user_b) pueden acceder
- **portfolio**: P√∫blico para lectura, solo el owner puede editar

## üìù Notas Importantes

1. **IDs:**
   - `profiles.id` es UUID (mismo que auth.users)
   - El resto de tablas usa `bigint` con secuencias

2. **Timestamps:**
   - Todas las tablas principales tienen `created_at`
   - `profiles` y `proposals` tienen tambi√©n `updated_at`

3. **Storage:**
   - Los archivos se guardan en Supabase Storage
   - Las tablas solo almacenan la ruta (`storage_path`)

4. **JSONB:**
   - `proposals.tags` usa JSONB para flexibilidad en etiquetas

5. **Validaciones:**
   - `rating` est√° entre 0-5
   - `total_payment` y `amount` son >= 0
   - `email` y `wallet_address` son UNIQUE

## üéØ Queries √ötiles para el Frontend

### Obtener propuestas con informaci√≥n del employer:
```sql

  p.*,
  prof.full_name as employer_name,
  prof.rating as employer_rating
FROM proposals p
JOIN profiles prof ON p.employer_id = prof.id
WHERE p.status = 'publicada'
ORDER BY p.created_at DESC;
```

### Obtener aplicantes de una propuesta:
```sql
SELECT 
  pa.*,
  prof.full_name,
  prof.rating,
  prof.bio
FROM proposal_applicants pa
JOIN profiles prof ON pa.freelancer_id = prof.id
WHERE pa.proposal_id = $1
ORDER BY pa.created_at DESC;
```

### Obtener mensajes de un chat con sender info:
```sql
SELECT 
  cm.*,
  prof.full_name as sender_name
FROM chat_messages cm
JOIN profiles prof ON cm.sender_id = prof.id
WHERE cm.thread_id = $1
ORDER BY cm.created_at ASC;
```
